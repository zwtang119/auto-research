# 将 MiMo Sprint 用作纯文件研究工厂

> **Date**: 2026-06-12
> **Audience**: 无编程背景的产品经理
> **Status**: 运营模型
> **Project**: CDS4WorldCup

## 1. 一句话定义

MiMo 纯文件研究工厂是一种工作流：MiMo 在本地机器上运行长程研究 Sprint，将每一个中间结果写为 Markdown 或 JSON 文件，然后交由人类审核、批准、拒绝或将其提升为正式项目产出。

该工厂不需要数据库、服务器、仪表盘或自定义后端。文件树就是产品、审计记录和操作界面。

## 2. 为什么用纯文件，而不是数据库或 institute-one

`institute-one` 是一个完整的 AI 研究所运行时：FastAPI 服务、SQLite 数据库、调度器、事件总线、Obsidian 仓库导出、前端和多个 CLI 代理。作为参考架构很有价值，但对当前 CDS4WorldCup 的运营模型来说过于笨重。

当前项目约束：

- 知识库是 Markdown。
- 不应引入数据库或在线服务。
- MiMo 输出是候选材料，不是最终事实。
- 公共页面不得暴露 MiMo、供应商名称或内部代理机制。
- 人工审核仍是决策关卡。

因此实际选择是：

- 借鉴 `institute-one` 的运营原则。
- 不引入其运行时。
- 将所有状态和输出保持为产品经理可检查的文件。

## 3. 工厂流程

工厂有五个阶段。

```text
Input -> MiMo Processing -> Candidate Output -> Human Review -> Official Promotion
```

| 阶段 | 负责人 | 发生什么 | 文件区域 |
|---|---|---|---|
| 输入 | 产品经理 | 启动 Sprint，陈述优先级，确认边界 | 仅 Prompt |
| 处理 | MiMo | 读取文档，检查队列，运行来源检查，起草候选材料 | `data/ops/**` |
| 候选产出 | MiMo | 生成结构化 Markdown / JSON 制品 | `candidate/`, `review_queue/`, `results/ops/` |
| 人工审核 | 产品经理 | 批准、修改、拒绝或要求来源检查 | `review_queue/` |
| 正式提升 | 人工 / 编码代理 | 将已批准材料转入正式数据、wiki、站点或卡片 | `wiki/`, `site/`, `data/processed/`, `artifacts/` |

关键规则：MiMo 可以准备工作，但不得自行发布正式事实。

## 4. 文件夹职责

### MiMo 操作区域

| 路径 | 含义 | 产品经理心智模型 |
|---|---|---|
| `data/ops/campaign_state.json` | 当前活动状态、最后检查点、活跃阻塞项、审核状态 | 工厂仪表盘 |
| `data/ops/task_queue.json` | 有序任务列表，含状态和优先级 | 生产排程 |
| `data/ops/candidate/` | 草稿研究材料 | 在制品货架 |
| `data/ops/review_queue/` | 等待人工审核的文件 | 质量控制队列 |
| `data/ops/mimo_outputs/` | 检查点、日志、诊断信息 | 运行日志 |
| `results/ops/` | Sprint 总结和运营报告 | 交接班报告 |

### 正式项目区域

| 路径 | 含义 | 为何 MiMo 不能自由写入 |
|---|---|---|
| `data/processed/` | 正式处理后数据 | 下游站点和分析依赖它 |
| `artifacts/team-cards/` | 正式球队路径卡片 | 必须通过来源和结构审核 |
| `wiki/` | 正式知识库 | Marginalia 格式和决策需要人工控制 |
| `site/` | 公共 GitHub Pages 站点 | 面向公众，品牌敏感 |
| `docs/design/specs/` | 正式规范 | 规范是执行参考，不是草稿纸 |
| `schema/`, `templates/`, `example/` | 协议定义 | 应保持稳定 |

## 5. MiMo 应该做什么

MiMo 最适合用于重复性研究劳动：

- 路径卡片审核
- 来源缺口审核
- 候选比赛上下文
- 候选压力卡片
- 赛后候选结算
- 候选赛后复盘
- 认知漂移报告
- 审核队列质量审计
- 正式变更提案
- 来源抓取诊断
- 下一 Sprint 任务展开

当工作涉及读取大量文件、比较结构化字段、整理缺失证据或准备审核材料包时，MiMo 尤其有价值。

## 6. MiMo 绝不能做什么

MiMo 绝不能：

- 未经人工审核发布正式事实。
- 将自身推理视为绿色来源。
- 未经明确授权修改正式文件。
- 生成投注建议或价值判断。
- 将新生成的 MiMo 数据重命名为 Kimi 数据或任何其他原始外部来源。
- 如果仍有安全工作可做，在完成单个交付物后即停止。
- 在仅一次抓取失败后就宣布外部来源不可用。

如果 MiMo 发现正式文件可能有误，应撰写正式变更提案，而非直接应用修改。

## 7. 人工产品经理职责

### Sprint 前

产品经理应决定：

- 这是长程 Sprint 还是短任务？
- 优先主题是什么？
- 是否允许正式变更？
- 之前的审核文件哪些已批准或拒绝？

默认为长程 Sprint，不允许正式变更。

### Sprint 期间

除非 MiMo 请求输入，否则不需要主动管理。

如果可以监控，仅检查：

- 它是否仍在写入检查点？
- 它是否停留在允许的文件夹内？
- 它是否遇到了需要人工判断的阻塞？

### Sprint 后

按以下顺序阅读：

1. 最新的 `results/ops/sprint-*.md`
2. `data/ops/review_queue/` 中的新文件
3. 然后才是 `data/ops/candidate/` 中的支撑文件

对于每个审核请求，在心中或在下一个 Prompt 中标记为：

- `approve`
- `revise`
- `reject`
- `needs-source-check`

不要仅因为写得好就提升材料。只有当来源级别和审核说明支持时才提升。

## 8. 标准 Sprint Prompt

### 首次或常规长程 Sprint

```text
请你执行 docs/ops/mimo-season-campaign.md。
这是本地 Mac mini 上的 24 小时长程研究 sprint。
请先恢复 campaign_state 和 task_queue，然后选择最高价值任务开始。
所有产物写入 candidate/review_queue/results/ops，不要修改正式页面、正式 wiki、spec、schema 或 git 状态。
完成单个 proposal/review request 后不要停止，回到 task_queue 继续执行下一个可安全任务。
```

### 审核后继续

```text
请继续执行 docs/ops/mimo-season-campaign.md。
我已审核上一轮 review_queue：<写明 approve/revise/reject/needs-source-check>。
请先把审核结果写入 campaign_state，再继续处理 task_queue 中下一项。
除非我写 SHORT_RUN_OK，否则不要在完成单个 deliverable 后结束。
```

### 指定主题优先

```text
请继续执行 docs/ops/mimo-season-campaign.md。
本轮优先处理：<比赛/球队/审计主题>。
仍然只写 candidate、review_queue、mimo_outputs 和 results/ops。
```

### 允许一次正式变更

仅在审核正式变更提案后使用。

```text
请继续执行 docs/ops/mimo-season-campaign.md。
APPROVE_FORMAL_MUTATION: <精确文件路径或目录>
本轮只允许修改上述路径。修改前先写 formal-change proposal，修改后运行 proposal 中列出的验证命令。
```

### 测试长程纪律

```text
请继续执行 docs/ops/mimo-season-campaign.md。
这是长程 worker loop，没有 SHORT_RUN_OK。
不要修改 data/processed、artifacts、site、scripts、tests、wiki 正式文件。
如果最高优先级任务因 APPROVE_FORMAL_MUTATION 缺失而阻塞，请执行 task_queue 中不依赖正式授权的安全任务。
少于 2 小时不得停止，除非写出 safe_work_exhaustion_matrix 并证明所有安全任务类别都不可执行。
```

## 9. 首轮运行的经验教训

早期 MiMo 运行产生了有用的候选材料，同时也暴露了运营风险。

观察到的优势：

- 能快速读取大量本地文件。
- 能生成结构化候选 JSON 和审核请求。
- 能识别来源缺口和下游影响。
- 当控制文档变得更加明确时，表现会改善。

观察到的风险：

- 除非边界明确，否则可能写入超出预期的文件夹。
- 可能将被阻塞的正式变更视为停止的理由。
- 可能完成单个交付物就宣布 Sprint 结束。
- 除非来源规则严格，否则可能将来源冲突变为事实断言。
- 可能使来源抓取失败听起来比证据支持的更广泛。

已在 `docs/ops/mimo-season-campaign.md` 中添加的当前防御措施：

- 使用 `git status --short` 的写入边界关卡。
- 明确的 `APPROVE_FORMAL_MUTATION` 机制。
- 正式世界杯事实的官方来源关卡。
- 在宣布来源被阻塞之前的抓取重试诊断。
- 正式变更被阻塞时的安全任务池。
- 每个任务输出的最低深度要求。
- 运行不足两小时的提前停止规则。

## 10. 三阶段路线图

### 第一阶段：稳定工厂

目标：MiMo 停留在边界内并产出可审核的输出。

行动：

- 所有输出保存在允许的 ops 文件夹中。
- 要求 Sprint 总结和审核请求。
- 将正式变更锁定在明确审批之后。
- 维护非空的任务队列，包含安全后备任务。
- 跟踪 Sprint 是否过早停止。

退出标准：

- 无写入边界违规。
- Sprint 总结列出所有变更文件。
- 审核请求包含来源政策说明和建议的人工操作。
- 短程运行包含真实的安全工作耗尽说明。

### 第二阶段：模板化工作

目标：重复性任务成为可预测的产品。

行动：

- 为路径卡片审核、来源缺口审核、比赛上下文候选、结算候选和认知漂移报告创建标准模板。
- 标准化必填章节：
  - 已检查的输入
  - 方法
  - 发现
  - 来源级别
  - 缺口
  - 后续行动
  - 审核说明
- 添加事件日志，如 `data/ops/events.jsonl`。
- 保持各 Sprint 间的 Prompt 一致性。

退出标准：

- 产品经理仅凭文件结构即可识别任务类型和质量。
- 不同 Sprint 对同一任务类型产出可比较的输出。
- 人工审核时间减少。

### 第三阶段：规模化比赛运营

目标：工厂在赛事期间支持比赛周期运营。

行动：

- 赛前准备比赛上下文。
- 为重要比赛准备压力卡片。
- 赛后准备结算候选。
- 审核结算后准备复盘候选。
- 仅将已批准的材料发布到正式输出。

退出标准：

- 一场比赛能在单个 Sprint 周期内完成从上下文准备到赛后审核的全流程。
- 每次公开更新都有清晰的审核轨迹。
- 候选材料在获批前不会泄露到公共页面。

## 11. 健康指标

### 工厂健康

| 指标 | 良好状态 |
|---|---|
| 写入边界违规 | 零 |
| Sprint 总结完整性 | 列出已完成任务、变更文件、阻塞项、审核队列 |
| 审核请求质量 | 包含建议操作、来源说明、风险说明、下一个 Prompt |
| 安全工作处理 | 被阻塞的正式工作转入安全后备任务 |
| 提前停止纪律 | 短程运行解释为何没有安全任务可做 |

### 研究质量

| 指标 | 良好状态 |
|---|---|
| 来源级别覆盖 | 每个事实性或候选声明都有绿色/黄色/红色标注 |
| 正式事实 | 球队名单、分组、赛程、场馆、比赛结果使用主要或存档来源 |
| 候选转批准率 | 已批准材料可追溯到审核请求 |
| 漂移报告 | 每隔几天总结认知偏差和重复错误模式 |
| 公共边界 | 公共站点不包含内部供应商术语或投注语言 |

## 12. 产品经理操作日常

每日或 Sprint 周期日常：

1. 使用标准 Prompt 之一启动或恢复 MiMo。
2. 让它运行，不要微观管理。
3. 先阅读最新的 Sprint 总结。
4. 审核待处理的审核请求。
5. 决定批准/修改/拒绝/需要来源检查。
6. 如需正式变更，仅批准精确路径。
7. 在包含审核结果的情况下启动下一个 Sprint。

产品经理的工作不是重写 MiMo 的产出。产品经理的工作是控制什么成为正式内容。

## 13. 最终运营原则

MiMo 应表现得像一个研究生产团队，而不是一个自主发布者。

它可以收集、比较、起草、分类和准备。由人类来决定什么足够真实、足够安全、足够有用，可以进入正式项目。
