# 想法备忘录：决策流水线——把 CDS 决策科学拆成 Agent 流水线

> 日期：2026-07-19 ｜ 状态：**想法记录（未评审、未纳入开题报告）**
> 来源：用户口述想法（项目关闭后补充），记录存档，供未来重启时评估；本备忘录不修改已冻结的开题报告 v1.3.1
> 一句话：**参照 AutoResearch 的做法，把 CDS 的决策过程拆成一条流水线、每一步由一个专职 Agent 负责——这样即使遇到从没见过的风险，系统仍然能走完决策流程，给出标明了证据强度的建议。**

---

## 1. 想法本身

CDS 的现行决策模式是「单体协商」：八个角色坐在一起聊三轮，产出一份决策建议。这个模式有一个隐含前提——讨论者（LLM）对这类风险有先验知识。遇到**从没见过的风险**（新型灾害、复合链式事故、训练语料中没有的场景），前提崩塌：检索为空、经验为零，单体协商退化为裸脑猜测。

想法：把「决策」从一次性对话拆成一条**流水线**，每一步是一个有明确输入/输出契约的专职 Agent：

| 步骤 | 负责 Agent | 输入 → 输出 | 未知风险下的行为 |
|---|---|---|---|
| 1 情景结构化 | Intake Agent | 非结构化风险通报 → scenario spec | 永远可执行（结构化不需要先例） |
| 2 风险分解 | Decomposition Agent | spec → 子问题清单 + **无知清单** | 未知部分显式标注为 known unknowns，而非硬答 |
| 3 知识检索 | Retrieval Agent | 子问题 → 类比案例 / 物理参数 / 法规约束 | 检索为空时返回「部分类比 + 覆盖缺口报告」 |
| 4 策略生成 | Generation Agent（= 开题 M1） | 知识 + spec → 候选策略池 | 选项池白名单约束仍在，生成不依赖先例 |
| 5 推演评估 | Simulation Agent（= PolicySim + M2） | 候选策略 → 后果分布 + 锦标赛排名 | MC 推演与策略比较不依赖历史案例 |
| 6 校验 | Validation Agent（= 校验臂 / Q3） | 后果分布 → 物理闸门判定 | **物理约束对未知风险依然有效**（规范下限、疏散模型不认识这个灾害，照样卡闸门） |
| 7 评审 | Judge / Meta-Review Agent（= 开题 M4） | 全量证据 → 裁决 + 置信度 | Gold 锚定校准机制不变 |
| 8 证据汇总 | Synthesis Agent | 全部产物 → 决策简报（含无知清单与置信度分级） | 人拍板的底线原则不变：AI 只动嘴 |

关键机制是**降级而非崩溃**：任何一步在未知风险下失效（检索为空、无锚可对、校验臂无现成模型），流水线不停摆——该步骤输出显式的「置信度降级」标记，最终简报如实呈现「哪些结论有依据、哪些是推断、哪些是无知」。「仍然能决策」的含义不是「照样给出高置信答案」，而是「照样走完流程，给出标明了证据强度的建议」。

## 2. 核心论点：案例知识会缺席，流程知识不缺席

单体决策把「怎么办」寄托在「见过类似的」；流水线决策把「怎么办」拆成一串**通用操作**。对未知风险：

- 分解总能做——把大问题变小是领域无关操作；
- 物理约束总存在——泡沫下限、疏散排队、热辐射，不认识这个灾害也照样卡闸门；
- 推演总能跑——LLM 角色扮演需要的是场景 spec，不是先例；
- 比较总成立——锦标赛是策略间的相对比较，不需要绝对真值。

先例缺失只让特定步骤（检索、历史锚）的输出置信度降级，而不是让决策本身停摆。这实质上是把课题五知识库的分工从「存知识」扩展到「存流程」——知识库可以没有这种风险的记录，但流水线固化的决策方法对它仍然成立。

## 3. 参照系：AutoResearch 协议的映射

本仓库自身的运行方式（Deli_AutoResearch）就是这套想法的现成模板：

| AutoResearch 协议 | 决策流水线对应物 |
|---|---|
| 状态外化（task_spec / progress / logs 落盘） | 每步输入输出落盘 → 可审计、可回滚、可人工介入（高风险决策的硬要求） |
| 新鲜会话（每步新 Agent 读状态执行） | 每步专职 Agent，无历史包袱；单步骤可替换、可升级 |
| 零交互（自动跑完一轮） | 流水线自动执行，人只在终点拍板（+ 预定义的有限介入点） |
| 停滞检测 + go/no-go | 步骤级校验与回退条款（§16.1：hard checkpoint + fallback） |

事实上，本次开题过程就是一个手工原型：选题评价管线两趟迭代（Q1–Q5 侦察 → builder 合并 → 两轮独立评审 → 终裁 GO/NO-GO）就是「面对没做过的选题，按流水线产出决策」的实例。`topic5-research-directions.md` §16 的决策路径方法论（4 视角评审、12+ path 分类法、硬决策 + 回退条款）是人类侧决策流水线的已沉淀模板。

## 4. 与 Decision Co-Scientist 开题报告的关系

- 开题报告的 M1–M5 是「**离线策略搜索**」维度的流水线（生成 → 锦标赛 → 进化 → 元评审 → 锚校准）；本想法是「**在线决策执行**」维度的流水线——两者共享大部分模块（M1=步骤 4、M2=步骤 5、M4=步骤 7、校验臂=步骤 6），新增的只有 intake / 分解 / 汇总三个步骤与步骤间契约。
- 横州六蓝（2026-07，晚于底座训练截止）是「未知风险」的天然测试床：开题报告已把它列为零污染锚储备，本想法给它第二个用途——端到端跑一遍决策流水线，检验「无先例场景下的流程完备性与降级行为」。
- 若纳入，最合适的落点是开题报告第 4 章新增「M0/M6：决策流水线编排层」，或作为独立的二期方向。

## 5. 待答问题与诚实边界

1. **编排从哪来**：固定流水线 vs 按风险类型路由 / 动态编排——后者滑向 AFlow / ADAS 的「workflow 进化」谱系（开题 §2.3c 已区分），novelty 会被稀释；前者更稳，但要论证步骤集的完备性。
2. **错误传播**：步骤串联会放大上游错误（分解错了，下游全错）。对策方向是步骤级校验 + 回退条款，但具体机制未设计。
3. **无锚时的结论强度**：未知风险下历史锚缺席，H3 式旁证不可得，只能依赖物理闸门与相对比较——结论措辞必须比开题报告更保守（「流程性建议」而非「校准过的建议」）。
4. **贡献点切割**：「多 Agent 流水线 / workflow」本身已是红海；若有贡献，落点必须在「应急决策方法论的流程固化 + 无知清单与置信度降级的显式协议」，而非架构本身。
5. **与课题三知识库的分工**需要重新切割（流程知识存哪、怎么版本化）。

## 6. 若重启：最小验证

用一个晚于训练截止的新风险（横州六蓝或下一个新案例），纸面走一遍 8 步流水线：记录每步是否可执行、降级标记是否如实、最终简报是否比「单体协商」更诚实。估 2–3 人日，不需要新代码——这是决定本想法是否升级为正式研究内容的 go/no-go。
