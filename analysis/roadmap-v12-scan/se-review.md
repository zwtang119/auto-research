## A. 不足清单（按严重度排序）

### A1. 第四旋钮加入后统计力账本未重算："18 格"已过时，实际 36 格
- **问题**：v1.1 §4.2 把信息集加为第四旋钮（roadmap:81），factorial 从 3×2×3=18 格变为 3×2×3×2=**36 格**；但 §4.3 仍沿用"18 格 factorial……每格仅 ~19"（roadmap:99）的旧账。R-B1（加旋钮）与 R-B2（补 power）在同一版本内相互打架却未对账。
- **后果**：power 缺口从"约一个数量级"恶化为约 20 倍；方案 A 的 10³–10⁴ 上限（10⁴/36 ≈ 278/格）已不足以支撑 36 格全确认性分析，方案选择被静默推向方案 B，但路线图未明说。
- **补救**：§4.3 按 36 格重算；或采纳 B 节建议把确认性旋钮砍至 2 个（8 格以内），使 10³–10⁴ 重新够用。二者必居其一，写入 OSF 前必须闭合。

### A2. 核心管线输入/输出未定义，B0 无法验收
- **问题**：B0 的四件交付物全部只有名字没有规格——(i) trace 记录管线：无 schema（字段、格式、版本）、无完整性校验规则；(ii) 冻结协议：prompt hash 全冻结后**由谁、何时、如何验证**（结算时重算 hash 比对？日志留存格式？）未定义；(iii) 任务流调度：选题频率、去重规则、聚簇控制（同日多场）未定义；(iv) 结算裁决：来源 (i)/(ii)/(iii) 三路径的裁定标准与截止日期缺失，而它是"论文的地基材料"（gap §5.1）。
- **后果**：B0 检查点（roadmap:167 "跑通 demo"）是定性口号，任何实现都可宣称通过；下游 B1 积累的数据可能因 schema 漂移而整批报废。
- **补救**：B0 出口增加四份最小规格文档（trace JSON schema + 冻结验证 SOP + 任务流调度规则 + 结算来源裁定记录），每份可一页内写完。

### A3. 无 pilot 阶段，B0→B1→B2 无 go/no-go 量化判据
- **问题**：时间线（roadmap:126–131）从"搭管线"直接跳到"数月积累"，中间没有小规模试跑；三个阶段的入口/出口判据全部缺席。唯一近似闸门的"达量"本身依赖未裁定的方案 A/B（A2 的连环债）。
- **后果**：管线缺陷（解析失败、聚簇失效、结算延迟）要到数月后达量时才发现，返工成本 = 整个 B1；这是典型的无试飞直接量产。
- **补救**：B0 与 B1 之间插入 B0.5 pilot：~50–100 次预测 × 小规模臂，go/no-go 判据例：解析成功率 ≥98%、trace 完整率 100%、结算延迟中位数 ≤72h、聚簇设计效应实测值落盘。pilot 通过才放行 B1。

### A4. 故障模式无监控/校验节点，静默失败面大
- **问题**：路线图没有任何运行期校验设计。已识别静默失败点至少五处：(i) 分身输出格式漂移 → 概率解析失败被静默丢弃，各臂存活率不等即引入选择偏差；(ii) 结算延迟/市场作废 → 样本悄悄缩水且非随机缺失；(iii) API 模型版本服务端变更 → "版本固定"承诺悄悄失效，prompt hash 冻结管不住 system 侧；(iv) 同日赛事聚簇 → 有效样本量低于名义量，power 账再度虚高；(v) 限流/超时导致部分分身缺失 → "300 分身"实际 n 逐次浮动。
- **后果**：任一发生都使预注册主张在不可见处失真，且期末考无法事后检测。
- **补救**：B1 加常驻监控节点（每次运行落盘：调用数/解析成功率/模型版本指纹/结算状态四字段），周级自动对账；缺失率超阈值即 fail-loud 停流——这与项目既有 fail-loud 纪律同构，成本极低。

### A5. 成本与资源无估算，方案 A/B 裁定缺一半输入
- **问题**：方案 A 的调用量粗算：10³–10⁴ 次已结算预测 × 至多 300 分身 × ≥2 模型家族 = **60 万–600 万次 API 调用**；辩论收敛臂多轮交互再乘 3–5 倍，达千万次级。按主流 API 定价量级估算为 **10³–10⁵ 美元**区间，外加数月墙钟时间。方案 B（1–2 主效应 + 其余探索性）调用量可压到方案 A 的 1/5–1/10。路线图对两者成本均无一行数字，"启动 B1 前裁定"（roadmap:101）因此缺少关键裁定依据。
- **后果**：可能在不知情中默认了贵 5–10 倍的方案；或 B1 跑到一半因预算/限流断供。
- **补救**：B0 内做一页成本模型（单价 × 调用量 × 臂数 × 冗余系数），作为方案 A/B 裁定的必备输入；在预算未落实前**默认方案 B**——统计上它本来就是 v1.1 的隐含归宿（见 A1），成本上它便宜一个量级。

### A6. 跨两篇的工程依赖未闭环：咬合句的措辞一致性无校验机制
- **问题**：检查点末条"双篇交叉引用核对"（roadmap:171）依赖人工记忆；§6 第 7 条已自承咬合单向脆弱（A 审稿删句则 B 叙事受损），补救（B 独立 OSF 预注册）写了但未设为默认动作。
- **后果**：低概率高影响，发生时才启动补救已晚（OSF 时间戳的价值在事前）。
- **补救**：把"B 独立 OSF 时间戳预注册"从补救升级为默认动作，咬合降级为加分项——一行改动消除单点故障。

## B. 过度设计清单

### B1. 四旋钮全确认性 factorial（36 格）
- **组件**：规模 × 人设 × 聚合 × 信息集四旋钮全做确认性分析（roadmap:76–81）。
- **为什么过度**：核心问题是"什么结构导致 herding vs 独立判断"，因果识别的**必要**旋钮只有两个——信息集（区分跟风复述与独立同答的唯一干净手段，M5）与群体规模（scaling 标题主张）。人设多样性与聚合规则是二阶问题；对它们做确认性断言是拿 4 倍样本量换非核心结论。
- **砍/降级建议**：确认性旋钮砍至 2 个（规模 × 信息集，3×2=6 格）；人设、聚合降为固定默认值 + 探索性附录。这正是方案 B 的精神，把它从"备选"扶正为默认设计。
- **砍后风险**：审稿人问"为什么不消融聚合规则"——以探索性结果 + 显式 MDE 声明回应即可；风险远小于 36 格全做导致的整体低 power。

### B2. commit-reveal 机制
- **组件**：Foresight Arena 式 commit-reveal（roadmap:90）。
- **为什么过度**：commit-reveal 解决的是**多方竞技场**中参赛者事后改答案的对抗性问题。Paper B 是单团队自建实验，对手方不存在；OSF 时间戳预注册 + prompt hash 日志 + git 提交链已提供同等强度的 ex-ante 证明。
- **砍/降级建议**：降级为"OSF 预注册 + 每日 hash 落盘（git log 即证据链）"，删除 commit-reveal 协议层。
- **砍后风险**：几乎为零；评审若质疑，git 不可变历史 + OSF 时间戳是学界标准答案。

### B3. "300 分身"作为规模旋钮的固定水平
- **组件**：规模水平 3/30/300（roadmap:78）。
- **为什么过度**：300 这个数来自 kimi 轶事复刻冲动，不是机制需要。规模收益曲线用对数间隔的小水平（如 3/10/30/100）即可识别拐点与负收益区；300 臂的成本是 3 臂的 100 倍，却只贡献曲线末端一个点。
- **砍/降级建议**：水平改为 3/10/30（或 3/30/100），写作中明示"300 是轶事值，本实验检验的是曲线形态而非复刻该点"。
- **砍后风险**：与 kimi 叙事的呼应变弱——但 §6 第 6 条已裁定认识论绝缘是永久的，呼应本来就只剩修辞价值。

### B4. 双人盲标 Cohen's κ ≥ 0.80 的人工归因编码
- **组件**：§4.4 双人盲标 κ≥0.80（roadmap:114）。
- **为什么过度**：同一节已规定"结论权重放在可自动复核的定量指标上，人工编码仅作补充"。对补充性材料套用主结论级的信度门槛（κ≥0.80 是范式对**承重**归因的要求），是人力镀金；且 herding 双层指标全自动后，人工编码的边际信息量很低。
- **砍/降级建议**：保留对称编码与预注册 taxonomy（便宜且防腐），人工编码降级为 illustrative case study，κ 报告为描述值不设硬门槛，或仅对抽样子集做单人编码。
- **砍后风险**：故事性证据变薄；若评审要求，可在 revision 阶段补做盲标（彼时结论已不依赖它）。

### B5. 期末考的随机分层抽样 + 双侧 power 分配
- **组件**：§4.5 随机分层划分 + 开发/评估各自 power 分配写入预注册（roadmap:120）。
- **为什么过度**：前瞻任务流自带时间轴，"最后到站的 X% 事件自动成为期末考"是零成本且防过拟合强度最高的划分（开发集结果永远无法泄漏进未来事件）。随机分层反而引入同源弱防护 + 域偏移论证义务 + 双侧 power 会计三层复杂度，去防一个时间划分天然不存在的问题。
- **砍/降级建议**：改回时间块留出（"B1 积累期满后的后续 N 场 = 期末考"），域偏移以如实披露处理；power 只对期末考单侧做账。
- **砍后风险**：赛事日历域偏移混杂（gap §5.4 的担忧方向反转）——但那是披露问题不是设计缺陷；复杂度省下来的是实实在在的。

### B6.（Paper A 附带）PIV 八字段全量标注
- **组件**：PIV 协议完整性向量八字段（经 gap 文档转引，v2:232）。
- **为什么过度**：审计对象只有 5 个预测者，8 字段 × 5 对象 = 40 个标注单元，字段间高度相关（可追溯/缺失/污染边界在 5 个对象上几乎共线）。对外部盲审 κ 而言，字段越多一致性越难过线，反而威胁 R-A2 这唯一升档杠杆。
- **砍/降级建议**：核心字段压至 3–4 个（存在性、可追溯性、污染边界），其余并为复合分。
- **砍后风险**："协议完整性作为一等字段"的颗粒度变细；对 Paper A 主主张无伤。

## C. 总体判断

**优化即可，不需要重构。** 骨架（前瞻受控实验 + OSF 预注册 + 信息集消融做因果识别 + kimi 降格为假设来源）在系统工程上成立且方向正确；全部缺口都是规格级问题——四份 I/O 规格、一个 pilot 闸门、一页成本模型、一处 36 格对账——补齐工作量在 1–2 周量级，不触及架构。真正的返工风险只有一个：若不加 pilot 与监控节点直接放量 B1，规格债会在数月后以数据报废的形式到期。
